# [제안서 초안] 상용 퍼블릭 LLM 연계형 Safe AI 구축 제안 (제안 1안)

PPT 작성을 위한 중간 산출물로서, 고객사 제안 요청(RFP) 항목에 맞춰 '외부 상용 LLM(ChatGPT, Gemini 등)과 Safe AI를 연동하는 방안'의 구조와 논리를 정리한 마크다운 문서입니다.

---

## 1. 공통 요청 사항 (Executive Summary)

### 1) 구축 구조 및 적용 범위
* **구축 구조:** 사내 업무/개발망과 외부 인터넷망을 잇는 경계인 **DMZ 구간에 'Safe AI (API Gateway Proxy)'를 배치**합니다. 사내 사용자와 사내 서버는 내부 IP 통신으로 제한된 DMZ 내의 Safe AI만 호출하며, 외부 API 통신은 오직 Safe AI가 대리(Proxying) 수행합니다.
* **적용 범위:** 전사 IT 개발팀(Cursor, Copilot 등 AI IDE 지원), 사내 시스템 개발 부서(RAG 기반 AI 챗봇 연동 API 지원), 신사업 기획팀 등 외부 LLM 연동이 필요한 전사 모든 조직에 공통 API 인프라로 적용됩니다.

### 2) 보안 및 운영 측면의 특징
* **보안 측면 (금융보안원 보안대책 평가 요건 충족):** 최근 금융당국의 '망분리 개선 로드맵'에 따른 생성형 AI 도입 시, 필수적으로 거쳐야 하는 **금융보안원의 보안성 평가(클라우드/SaaS 이용 심사)**를 통과하기 위한 핵심 기술 요건을 제공합니다. 여신/수신 고객의 고유식별정보(상담내역, 신용정보 등 PII)가 퍼블릭으로 나가지 않도록 하는 **사전 마스킹(Masking)** 처리 및 전자금융감독규정 기준의 **100% 프롬프트 감사(Audit) 로그**를 통해 규제 특례(샌드박스) 승인을 완벽히 지원합니다.
* **운영 측면:** 중앙 집중형 토큰 예산/비용 통제(부서별 Quota 대시보드), 유사 질의에 대한 시맨틱 캐싱(Semantic Caching)을 통한 원가 절감. 다기종 외부 LLM 도입 시 발생하는 API 연동 규격 파편화 완전 해소.

### 3) 필요 솔루션 및 인프라 규모 (초기)
* **필요 솔루션:** Safe AI Proxy Gateway Core, 관리자 관제/통계 대시보드(Web), 로컬 PII 필터링 모듈, Redis(캐싱/토큰 제어 처리용), PostgreSQL(감사/정책 메타 정보 처리용).
* **인프라 규모 (500명~1,000명 동시접속 기준):** 
  * 고가의 GPU 서버 일절 불필요 (LLM 연산은 외부 퍼블릭 클라우드 활용).
  * WEB/WAS (Proxy 역할): 4vCPU / 16GB RAM x 2대 (HA 이중화 구성)
  * DB / Cache Node: 8vCPU / 32GB RAM x 2대 (DB 이중화)
  * NAS/SAN 스토리지: 최소 1TB (감사 로그 보관 기간에 따라 확장)

### 4) 예상 장단점 및 고객사에 적용 시 좋은 점(시너지)
* **장점:** 즉시 도입 속도가 가장 빠름. GPT-4, Gemini 등 세상에서 가장 뛰어난 성능의 LLM을 바로 실무에 도입. 사내에 무거운 AI 인프라 유지보수(GPU 운영) 부담 없음.
* **단점 및 극복 방안:** 로컬(On-Premise) 대비 데이터가 외부 클라우드로 넘어가므로 '금융권 클라우드 서비스 이용 (CSP) 심사' 및 보안성 검토라는 행정적 허들이 존재합니다. 하지만 Safe AI가 제공하는 "신용정보/PII 자동 마스킹"과 "100% 감사 로깅" 기능이 이 심사 통과를 위한 가장 강력한 기술적 소명 자료가 됩니다.
* **시너지 파급효과:** 캐피탈사의 가장 큰 딜레마인 **"금융 망분리 준수"**와 **"비즈니스/개발 조직의 최신 AI 도입 요구"** 간의 충돌을 프록시 계층 하나로 완전히 해결합니다. 경쟁 캐피탈사 대비 월등히 빠른 디지털 전환(DX) 속도와 차별화된 금융 서비스를 런칭할 기틀을 확보합니다.

## 2. 제안별 세부 내용 (제안 1안. 상용 LLM + Safe AI 연계)

### 1) 제안별 아키텍처 구성도 & 설계방안
망분리 원칙을 훼손하지 않고 외부 Cloud LLM을 안전하게 호출하는 '브릿지 아키텍처'를 설계합니다.

```mermaid
graph TD;
    subgraph 사내 업무/개발망 (Internal Network)
        Devs[사내 개발자 / IDE] --> |내부 통신| InternalLB[내부 Load Balancer]
        InternalApps[사내 AI 서비스 서버] --> |내부 통신| InternalLB
    end

    subgraph DMZ 구간 (망연계 및 보안 통제)
        InternalLB --> SafeAI[Safe AI MVP 프록시 서버]
        
        subgraph Safe AI 내부 핵심 로직
            SafeAI --> Auth[1. 부서/SSO 권한 검증 및 Quota 확인]
            Auth --> Masking[2. DLP & 개인정보 마스킹 처리]
            Masking --> Caching[3. 시맨틱 캐싱 히트 판별]
            Caching --> AuditLog[(4. 감사 로그 전체 저장)]
            Caching --> Keys[5. 외부 API Key 맵핑 체결]
        end
    end

    subgraph 외부 인터넷망 (Public Cloud)
        Keys --> |Whitelist 방화벽 오픈 후 HTTPS 전송| Firewall[외부망 통신 방화벽]
        Firewall --> OpenAI[OpenAI API 서버]
        Firewall --> Anthropic[Anthropic API 서버]
        Firewall --> Gemini[Google Gemini API 서버]
    end
```

### 2) Safe AI 솔루션 역할 및 보안/망분리 환경을 고려한 설계 방안 특장점

**① 망분리/DMZ 특화 "단일 보안 출입구 확보 (Single Egress)"**
사내의 무수히 많은 시스템과 개발자가 개별적으로 방화벽 열람/우회를 신청할 필요가 없습니다. 회사의 외부 방화벽은 오직 DMZ에 위치한 'Safe AI 서버 주소'에서 나가는 AI 트래픽만 허용(Whitelist)하도록 심플하고 안전하게 재편됩니다. 네트워크 보안 책임자의 관리 부담이 100배 줄어듭니다.

**② 비인가 자격 증명 유출 원천 차단 (Vault Secret Management)**
개발팀이 직접 퍼블릭 클라우드에서 API Key를 발급받고 소스코드에 하드코딩하면, GitHub 유출 등으로 대형 사고가 발생합니다. 본 구성에서는 사내 개발자는 오직 사내용 SSO 토큰(Safe AI용 인증서)만 발급받고, 실제 막대한 요금이 결제되는 진짜 API Key(OpenAI Key 등)는 DMZ 구간의 Safe AI 내부 금고에 은닉되어 절대 사내/외부로 노출되지 않습니다.

**③ DMZ 구간 격리 원칙 통한 "Inbound 역류" 원천 봉쇄**
프록시는 외부로 향하는 연결(Outbound)에 대한 세션만 맺습니다. 외부망(퍼블릭 LLM)에서 사내망으로 먼저 연결을 시도하는 Inbound 포트는 원천 차단된 구조로 설계되어, 외부 모델 해킹 시 사내 망으로 번지는 횡적 이동(Lateral Movement) 위험이 0에 수렴합니다.

### 3) 제안별 필요 H/W 장비 구성 요약
*(1. 공통 요청 사항의 내용과 갈음)*
본 구축안의 핵심은 사내에 막대한 비용을 초래하는 NVIDIA GPU 장비 없이, 기존 레거시 웹/WAS 인프라(일반 CPU/RAM 구조) 장비 최소 4대만으로 고성능 엔터프라이즈 AI 허브를 런칭할 수 있다는 점입니다. 

### 4) 사례별 장점 및 단점
* **장점 요약:** 압도적 퍼포먼스(최신 GPT 연동), 최소한의 초기 H/W 투자비용(CAPEX 절감), 망분리 상태에서 개발 생태계(Cursor 등)의 완전한 프리패스 지원.
* **단점 요약:** 퍼블릭 클라우드로의 데이터 이동에 따른 금융감독원/금융보안원의 망분리 완화 및 클라우드 이용 보안성 심사를 거쳐야 하는 행정적 허들 존재.

### 5) 캐피탈사 적용 사례 시나리오 (Expected Use Cases)
* **사례 1 (사내 IT 개발 혁신 - 망분리 환경의 IDE 연동):** 망분리 규제로 단절된 내부 개발망에 Safe AI 프록시를 연동하여, 코어뱅킹/대내외 채널 시스템 개발자들이 Cursor, Copilot 등의 AI 코딩 어시스턴트를 합법적이고 안전하게 사용. 신규 비즈니스 로직 런칭 속도 대폭 단축.
* **사례 2 (여신 심사 및 고객 상담 요약 RAG 연동):** 방대한 고객 상담 녹취록이나 복잡한 법인/개인 여신 심사 서류를 요약하는 사내 AI 챗봇 구축 시, Safe AI를 필수 관문으로 사용. 고객의 민감 신용정보(계좌, 대출 잔액 등)가 외부 LLM에 넘어가기 전 Safe AI가 자동으로 마스킹(임의 기호 치환) 처리하여 신용정보보호법 완벽 준수.
